Case study · for the Philips brief · Lens 02 · SDK debugging

A silent iOS data-loss bug, traced to the initialisation sequence.

An iOS undercount of Application Opened events vs. Android — root-caused to a Segment SDK 1.9.1 gate that silently drops events when enabled = false.

/ Major quick-service restaurant · Segment · analytics-swift 1.9.1

Why this is in the Philips selection
The kind of failure mode that matters when clinical or operational data has to be trustworthy. An SDK-level gate silently dropping cold-launch events looks fine on every dashboard until you compare platforms. Telemetry that undercounts is worse than telemetry that's missing — the dashboards stay green while the truth degrades quietly. This case is the SDK-debugging discipline behind catching it: comparing platforms, reading the SDK source, finding the gate, fixing the sequence.
→
What happened
A major QSR's systematic iOS undercount of Application Opened events versus Android was root-caused to a Segment SDK 1.9.1 gate in the app's initialisation sequence (process(incomingEvent:) returns immediately when enabled == false, silently dropping cold-launch events fired before remote config flipped the gate). Same engagement surfaced cross-platform tracking-plan parity gaps and a GTM misconfiguration where two international locales were feeding the main analytics container. Telemetry that undercounts is worse than telemetry that's missing — this is the SDK-level discipline behind catching it.
Cross-platform parity failures almost always trace back to startup-order assumptions inside a vendor SDK — the kind of detail that only shows up at runtime, on cold launch, in one OS. The fix isn't reading the docs. It's reading the SDK source, version-pinned to what actually ships, until the silent drop becomes visible.
Context
A major quick-service restaurant's mobile analytics showed a systematic gap between iOS and Android Application Opened counts. The bug was silent: no errors, no dropped-event logs — just a stable undercount. Downstream funnel reading was being skewed and nobody was sure why. The engagement also raised broader concerns about cross-platform tracking-plan parity and GTM container governance.
What I did
SDK-level debugging. Traced the iOS undercount through the app's initialisation sequence: rawSegment.enabled = false at startup, re-enabled only after remote config completed. Identified that Segment SDK 1.9.1 process(incomingEvent:) returns immediately when enabled == false — cold-launch events were dropped, not queued. Surfaced the same engagement's cross-platform parity gaps (web on legacy dataLayer/GA naming; apps on a unified Title Case Segment model) and the GTM misconfiguration where two international locales were feeding the main analytics container.
