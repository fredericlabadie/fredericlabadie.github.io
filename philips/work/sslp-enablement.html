Case study · Path 05 · Adoption & cultural transition

The reporting layer survived the people who built it.

A BigQuery, Dataform, and Looker Studio reporting build replaced recurring manual report pulls, then a structured handoff made the system maintainable by the client team.

/ State-regulated retail platform · analytics architect · 2025–2026

Role-fit signal
Adoption is part of the telemetry system. The Philips role asks for data storytelling, data literacy, and the cultural transition toward evidence-based product management. This case is the handoff layer: documentation, training, dashboards, and operating practices designed so the team can use the measurement system without the original builder in the room.
→
What happened
A state-regulated retail platform moved from two analysts spending the equivalent of nearly a combined work-week per month manually pulling data to a unified BigQuery, Dataform, and Looker Studio stack with structured enablement behind it. The visible metric was 38 analyst hours recovered per month. The stronger proof was durability: when the client team turned over, new analysts required zero ramp-up time on the data infrastructure.
The deliverable was not just a dashboard. It was independence. The final measure of an analytics engagement is whether the team changes how they work after the builder leaves.
Situation
Web, app, Google Search Console, Braze, and additional reporting data lived in disconnected sources with no shared infrastructure. A monthly reporting cycle required manual extraction, reconciliation, and formatting. Domain specialists had no clean self-serve path into their own data.
Task
Build the system and the conditions for adoption: unified data models, dashboards, documentation, repo access, branching and review practices, data dictionaries, ERD summaries, and training sessions built for the analysts who would inherit the system.
Action
Unified web, Search Console, Braze, and third-party reporting data into BigQuery via scheduled Dataform pipelines; built Looker Studio dashboards for cross-channel visibility; added Python automation against the Braze REST API; then documented the system for a client analyst audience, not an agency audience.
Result
The recovered hours were valuable. The stronger proof was that those hours did not snap back when the team changed. New analysts could start from domain expertise on day one because the documentation and training architecture had been built for the team that would inherit it.

Before
Two dedicated analysts. One monthly report. Nearly a combined work-week of manual data pulling per cycle.
Web, app, Search Console, and Braze data lived in disconnected sources. Each monthly cycle required manual extraction, reconciliation, and formatting.
Build
BigQuery, Dataform, and Looker Studio created the maintained reporting layer.
Domain specialists gained direct self-serve access to their own data. A Braze REST API automation removed another recurring CRM reporting task.
Handoff 1
Web: GTM structure, GA4 event taxonomy, Looker Studio architecture, BigQuery pipeline.
Documented for a client analyst audience. Operational clarity over technical performance.
Handoff 2
App and reporting: SDK patterns, GA4 DebugView, BigQuery schema, production SQL walkthrough.
Repo access, branching strategy, and PR review process were explained so the client team could validate new events independently.
Handoff 3
CRM: Braze dashboard orientation, Python automation walkthrough, API key rotation, checklist.
Confluence knowledge base, ERD summary, and data dictionary made the system legible to the inheriting team.
Adoption test
Team turnover. New analysts onboarded. Zero ramp-up time on data infrastructure.
