Case study · for the Philips brief · Lens 05 · Adoption & Cultural Transition

The system survived the people who built it.

Greenfield UX telemetry across web, app, and CRM — then a structured handoff designed so the client team could maintain and extend it independently.

/ State-regulated retail platform · analytics architect · 2025–2026

Why this is in the Philips selection
Same shape as enabling internal Philips teams to own their measurement. Documentation built for the team that inherits the system, not for the team that built it. Zero ramp-up after turnover is the right benchmark for any handoff into a permanent organisation — the test of "does the system survive the people who built it?" applies whether the inheriting team is a different consultancy, a client analytics function, or an internal Philips squad picking up an Adoption & Insights workstream.
→
What happened
A state-regulated retail platform's analytics function moved from two analysts spending the equivalent of two combined work-weeks per month manually pulling data to a unified BigQuery + Dataform + Looker Studio stack with three structured handoff sessions behind it. 38 analyst hours recovered per month, redirected from data-pulling to analysis. The final adoption test: when the team turned over, zero ramp-up time was required on the data infrastructure.
The final measure of an analytics engagement is whether the client team changed how they work as a result. A measurement layer that produces reports nobody reads has not been adopted. The 38 hours recovered per month is not the point — the point is what that recovered time was redirected to.
Situation
Two dedicated analysts spending the equivalent of two combined work-weeks per month on manual data extraction, reconciliation, and formatting for a single monthly report. Web, app, GSC, and Braze data lived in disconnected sources with no shared infrastructure. Domain specialists had no clean access to their own data.
Task
Unify the data layer so domain specialists could self-serve, free the analysts from manual pulling, and — the harder part — design the handoff so the system would survive the team that inherited it. Adoption was the real deliverable; the pipelines were the substrate.
Action
Unified GA4, GSC, and Braze into BigQuery via scheduled Dataform pipelines; built Looker Studio dashboards for cross-channel visibility; added a Python automation against the Braze REST API to remove another ~2 hours/month of manual CRM reporting. Then three structured handoff sessions (Web, App, CRM) — documented for a client analyst audience, not an agency one. Confluence knowledge base, ERD summary, data dictionary, repo access with branching strategy and PR review process explained, API key rotation, handoff checklist. Detail in the timeline below.
Result
The system survived the people who built it — the real test of an enablement project, and the headline above the metric. When the client team turned over, new analysts started from domain expertise on day one because the documentation and training architecture had been built for the team that would inherit it, not for the team that shipped it. The recovered analyst hours shifted permanently to analysis; they didn't snap back when ownership changed.

Before
Two dedicated analysts. One monthly report. Two combined work-weeks of manual data pulling per cycle.
Web, app, GSC, and Braze data lived in disconnected sources with no shared infrastructure. Each monthly cycle required manual extraction, reconciliation, and formatting. Domain specialists had no clean access to their own data.
Build
GA4, GSC, and Braze unified into BigQuery via scheduled Dataform pipelines. Looker Studio dashboards for cross-channel visibility.
Domain specialists gained direct self-serve access to their own data. The two dedicated analysts recovered 38 hours per month. A Braze REST API automation eliminated a further 2 hours/month of manual CRM reporting.
Handoff · Session 1 — Web
GTM container structure, GA4 event taxonomy, Looker Studio architecture, BigQuery web pipeline — end to end.
Documented for a client analyst audience, not an agency one. Operational clarity over technical depth. Every procedure needed to run a monthly cycle independently, written for the team that would inherit it.
Handoff · Session 2 — App
Firebase SDK patterns, GA4 DebugView for QA, BigQuery app pipeline schema, production SQL walkthrough line by line.
Repo access granted, branching strategy and PR review process explained. The client team could validate new events before production deployment without an external dependency.
Handoff · Session 3 — CRM
Braze dashboard orientation, Python automation walkthrough, API key rotation, handoff checklist.
Confluence knowledge base, ERD summary, and data dictionary. Everything documented to support a team that had never owned this infrastructure before.
Adoption test
Team turnover. New analysts onboarded. Zero ramp-up time on data infrastructure.
The final test of any analytics engagement: does the system survive the people who built it? The documentation and training architecture meant new team members could start from domain expertise on day one.
