Case study · Path 02 · GTM instrumentation

All analytics in the tag manager. Not one line in the site code.

A hardware authentication brand’s security requirement — no analytics code in the site codebase — forced the entire measurement layer into GTM. DOM-driven variables, history-state SPA pageviews, enhanced ecommerce, and full form fallout tracking, all from a container.

/ B2B/B2C hardware authentication brand · e-commerce instrumentation · 2016

Role-fit signal
Security forced the analytics layer out of the site codebase entirely. The Philips role describes translating data needs into event specifications, implementation requirements, and production telemetry. This case shows what that looks like under a hard constraint: a security requirement that removed the option to embed analytics in the site code forced cleaner architecture — versioned container, isolated deployment, no coupling between site releases and measurement layer.
→
What happened
A hardware authentication brand needed full e-commerce analytics for a Magento storefront, but a security policy required that analytics code live entirely outside the site codebase. Every tag, variable, and trigger was built in GTM. When the purchase funnel became a Single Page Application mid-build, the constraint stayed fixed. DOM-based variable schemas, wildcard tag matching, and history-state listeners handled the SPA. Enhanced ecommerce and full form fallout tracking shipped from the container alone.
The security requirement removed the option to embed. GTM became the complete measurement system — not a supplementary layer.
Context
The client was iterating rapidly on a Magento storefront with a custom frontend. Class names and element structures changed across sprints. A security concern about analytics code introducing codebase vulnerabilities required all measurement to stay in GTM — never in the site’s code. When the purchase funnel transitioned to a SPA mid-build, the DOM-based tagging approach had to adapt without the security constraint changing.
Task
Full implementation: write tagging requirements, build GTM containers, design DOM-based variable schemas, handle SPA pageview capture via history-state listeners, configure enhanced ecommerce across the purchase funnel, specify and implement form fallout and error tracking, QA against sprint releases.
Action
Built two GTM containers — one for the existing site, one for the new section. DOM-driven variable schemas captured element state without requiring stable class names. Wildcard tag logic and history-state change listeners replaced standard pageview triggers once the SPA appeared. Enhanced ecommerce implemented across product impressions, add-to-cart, checkout steps, and purchase events. Form fallout coverage across purchase and account forms.
Result
Zero analytics lines in the site codebase at any point. Enhanced ecommerce and form fallout live. SPA pageviews firing correctly via history state. As a side effect: adding IDs and stable class names for GTM targeting also improved WCAG compliance across the elements that needed them.
