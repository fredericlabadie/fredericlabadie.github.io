Case study · for the Philips brief · Lens 02 · Error taxonomy

Clean taxonomy, trustworthy alerts, enriched context.

Three layers of remediation on an alert channel where genuine checkout failures were buried in noise from low-traffic Active% triggers.

/ Major quick-service restaurant · Amplitude monitors

Why this is in the Philips selection
The Philips brief explicitly names error taxonomy and alert engineering. This is the case behind that claim — the hierarchical category IDs, threshold redesign, and the principle that error events must carry the same context as their success counterparts. The discipline transfers directly to clinical telemetry, where an alert that doesn't carry enough context to action is worse than no alert at all.
→
What happened
A noisy Amplitude alert channel where genuine checkout failures were buried in low-traffic Active% noise gained three layers of remediation: redesigned thresholds with troubleshooting guidance on each chart, a hierarchical error-category ID scheme (SYS-INF-001, SYS-APP-VAL-010) with governance rules against schema drift, and cart_id enrichment so error events carry the same context as their success counterparts. Alerts became trustworthy enough to drive action without detective work — and the taxonomy directly reduces false positives in the AI alert-triage system.
Error events must carry the same context as their success counterparts. Otherwise alerting is correct but actionable only with detective work — and on-call learns to silence the channel.
Context
An Amplitude alert channel at a major QSR firing on too many low-signal events — Active% triggers from low-traffic monitors drowning out genuine checkout failures. Error events did not carry the context their success counterparts had, which meant every alert kicked off detective work just to know which basket had failed. The alert engineering needed to be made trustworthy, and the schema needed to stop drifting.
What I did
Three layers (detail in the flow below). Threshold redesign — absolute rate + rate-of-change, with engineering-facing troubleshooting guidance directly on each chart so the alert itself carries the start of the diagnosis. Hierarchical error-category ID scheme (SYS-INF-001, SYS-APP-VAL-010) with deterministic mapping rules and governance guidelines against schema drift. Context enrichment — added cart_id to Checkout Started Error Received so engineering could correlate specific basket instances to checkout failures.
01
Threshold redesign
Redesigned alert thresholds (absolute rate + rate-of-change) and added engineering-facing troubleshooting guidance directly on each chart, so the alert itself carries the start of the diagnosis.
02
Error-category ID scheme
Designed a hierarchical, stable error-category ID scheme (SYS-INF-001, SYS-APP-VAL-010) with deterministic mapping rules and governance guidelines to prevent schema drift over time.
03
Context enrichment
Added cart_id to Checkout Started Error Received events so engineering could correlate specific basket instances to checkout failures — same context level as success events.
