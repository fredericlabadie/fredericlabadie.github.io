Case study · for the Philips brief · Lens 02 · Instrumentation

From "what should we measure?" to 30+ shipped tickets.

Greenfield mobile UX-telemetry layer across iOS and Android — spec to production, with cross-platform parity and consent built in.

/ State-regulated retail platform · analytics architect · 2025–2026

Why this is in the Philips selection
Direct mapping to the UX Telemetry Expert instrumentation responsibilities. Cross-platform telemetry architecture with consent management built in from the start (TrustArc, OneTrust). The shape of the work — schema design, SDK implementation, QA, production deployment with privacy guardrails — is the same shape Philips needs across its product estate. The state-regulated context here forced consent-aware tagging discipline that translates directly to a clinical/regulated environment.
→
What happened
A state-regulated retail platform's mobile app shipped to production with a full UX-telemetry layer underneath it — 30+ tickets authored, owned, and deployed across iOS and Android, cross-platform schema parity audited and closed, GA4 e-commerce taxonomy live, and consent management integrated via TrustArc + OneTrust before launch. The dashboards that now run on it work because this layer does.
The deliverable wasn't a dashboard. It was the analytics layer the dashboards eventually run on — taxonomy, schema, SDK implementation, QA, and production deployment across both platforms, with consent governance integrated before launch rather than retrofitted after.
Situation
A state-regulated retail platform building a brand-new mobile app from scratch. Zero prior instrumentation — no taxonomy, no schema, no SDK integration, no production deployment story for telemetry. Downstream marketing, product, and compliance work all depended on it being right the first time.
Task
Stand up the full mobile UX-telemetry layer end to end across iOS and Android — spec to production — with cross-platform parity audited explicitly and consent management built in from the start (state-regulated context: TrustArc + OneTrust, no after-the-fact bolting). The deliverable was the layer itself, not a deliverable about the layer.
Action
Designed and shipped the full GA4 e-commerce taxonomy (purchase, add_to_cart, view_item, begin_checkout, view_cart, refund with required items array, value, currency, transaction_id), screen-view coverage across core/auth/permission/system-state screens, interaction telemetry (bottom-nav, search-query submission with results count, PDP product metadata on every event, barcode-scanner with full payload, account-management flows, retention cohorts in GA4 Explore), and a governance layer around it (GA4 DebugView for real-time event validation, real-device testing iOS + Android, regression QA checklists per sprint, privacy-compliance review before any PII-adjacent property was specced, data-governance sign-off from client stakeholders). Detail in the three-card breakdown below.
Result
The mobile app launched with a UX-telemetry foundation the analytics team could trust on day one — and that downstream marketing and product work could build against without retroactive remediation. The cost of not doing the consent and schema work pre-launch would have been six to twelve months of reconciliation after the fact: the kind of debt that quietly disqualifies the data later when product asks for an attribution model or a cohort cut.
