// Case study / Lens 05 / Adoption & Cultural Transition

The system survived the people who built it.

Greenfield UX telemetry across web, app, and CRM, followed by a structured handoff designed so the client team could maintain and extend it independently.

// 01 / thesisadoption / independence / handoff

Core argument

The deliverable was not a dashboard. It was independence.

The final measure of an analytics engagement is whether the client team changes how they work after the builder leaves.

What happened

A manual reporting cycle became a maintained analytics system.

A state-regulated retail platform moved from two analysts spending the equivalent of two combined work-weeks per month manually pulling data to a unified BigQuery, Dataform, and Looker Studio stack with structured enablement behind it.

The visible metric was 38 analyst hours recovered per month. The more important result was durability: when the client team turned over, new analysts required zero ramp-up time on the data infrastructure.

Situation

The data existed, but ownership was trapped in manual work.

Web, app, Google Search Console, and Braze data lived in disconnected sources with no shared infrastructure. A monthly reporting cycle required manual extraction, reconciliation, and formatting. Domain specialists had no clean self-serve path into their own data.

Task

Build the system and the conditions for adoption.

The work was not only to unify the data layer. It was to design a handoff that would survive team turnover: clear documentation, repo access, branching and review practices, data dictionaries, ERD summaries, and training sessions built for the analysts who would inherit the system.

// 02 / enablement architecturebuild / document / transfer

Handoff design

Adoption was engineered into the project plan.

The pipeline mattered because it changed the work. The handoff mattered because it kept the change from depending on one person.

1Web handoff

GTM container structure, GA4 event taxonomy, Looker Studio architecture, and BigQuery web pipeline from end to end.

2App handoff

Firebase SDK patterns, GA4 DebugView QA, BigQuery app schema, and line-by-line walkthrough of production SQL.

3CRM handoff

Braze dashboard orientation, Python automation walkthrough, API key rotation, and operational checklist.

4Ownership layer

Confluence knowledge base, ERD summary, data dictionary, repo access, branching strategy, and PR review process.

Action

The reporting layer was redesigned around self-service.

I unified GA4, GSC, and Braze into BigQuery via scheduled Dataform pipelines; built Looker Studio dashboards for cross-channel visibility; and added a Python automation against the Braze REST API to remove another small but persistent manual CRM reporting task.

Then I documented the system for a client analyst audience, not an agency audience. The goal was operational clarity: what runs, where it lives, how to validate it, and what to do when ownership changes.

// 03 / resulthours recovered / turnover test / durable adoption

Result

The system passed the turnover test.

The recovered hours were valuable. The stronger proof was that those hours did not snap back when the team changed.

Outcome

The client team could work from domain expertise on day one.

  • 38 analyst hours recovered per month, redirected from data pulling to analysis.
  • Three structured handoff sessions across web, app, and CRM layers.
  • Zero data-infrastructure ramp-up when the full client team rolled off in June 2025 — incoming analysts were operational on day one.
  • System has been running independently since March 2026, when the build team transitioned to a new engagement. No escalations back.
  • Self-serve access improved for domain specialists who previously depended on manual reporting cycles.

Design lesson

A handoff is part of the architecture.

Documentation, repo practices, and training design are not administrative leftovers. They are the adoption layer of the analytics system. Without them, the pipeline works only while its builder is nearby.
