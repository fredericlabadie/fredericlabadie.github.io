Replatformed the CRM. The campaigns did not notice.

A full Oracle Responsys to Bloomreach CDP migration where the customer data model, event schemas, segmentation, and triggered-campaign architecture were redesigned instead of copied forward.

// 01 / thesismigration as model redesign

Core argument

The migration was the architectural opportunity.

Marketing platform migrations fail when they either break continuity or copy yesterday's data model into tomorrow's platform.

What happened

The CRM moved platforms without campaigns noticing the cutover.

A consumer-electronics retailer moved from Oracle Responsys to Bloomreach CDP with the customer data model redesigned for the new platform's primitives, not lifted from the old schema. The migration preserved zero campaign downtime through cutover.

Year-one post-migration results were +10.8% CTR YoY and +5.8% open rate YoY. The lift validated the architecture decision, not just the platform change.

Situation

The old platform shaped the old data model.

The retailer was running lifecycle marketing on Oracle Responsys, with a customer data model shaped by the platform's constraints: static list-based segmentation, identity stitching glued on, and consent state managed sideways. The decision to move to Bloomreach created a risk: a fresh copy of the old architecture in a new house.

Task

Preserve continuity while refusing to port the old model forward.

Two outcomes had to hold simultaneously: no campaign downtime through cutover, and no uncritical port of the Responsys-era data architecture. The migration needed to become a data-model redesign, not a platform swap.

// 02 / architectureidentity / events / campaigns / consent

Redesign

Build the model around the new platform's primitives.

Identity, consent, lifecycle events, and campaign triggers were treated as architecture, not migration afterthoughts.

1Customer data model

Redesigned from scratch for Bloomreach CDP primitives, with identity stitched across email, SMS, and on-site behavioural events.

2Event schemas

Lifecycle event taxonomy authored from product, transactional, and engagement signals.

3Segmentation

Segmentation logic rewritten to be queryable rather than enumerated: cohorts defined by behaviour, not static list membership.

4Triggered campaigns

Triggered-campaign automation built on new lifecycle event tracking with GDPR-compliant consent maintained through cutover.

Action

The platform migration became a lifecycle architecture rebuild.

I rebuilt the customer data model for Bloomreach, authored the lifecycle event taxonomy, rewrote segmentation logic around behavioural cohorts, and supported triggered-campaign automation on the new event layer while maintaining GDPR-compliant consent during the cutover.

The same engagement also included managing a team of two designers on the campaign-creative side: real, scoped people-management rather than a broad people-leadership claim.

// 03 / resultcontinuity / lift / durable model

Result

The campaigns stayed live. The model changed underneath them.

The goal was not merely to survive a cutover. It was to leave the CRM on infrastructure designed for what it now needed to do.

Outcome

The team could speak Bloomreach instead of translating from Responsys.

  • Zero campaign downtime during the platform cutover.
  • +10.8% CTR YoY in the first year post-migration.
  • +5.8% open rate YoY in the first year post-migration.
  • Customer identity stitched across email, SMS, and on-site behavioural events.
  • Consent state carried through the model rather than bolted on after migration.

Design lesson

A migration is a rare chance to refuse inherited architecture.

The safest migration is not always the one that copies the old system most faithfully. Sometimes the lower-risk path is to preserve operational continuity while redesigning the data model underneath, so the new platform is not forced to behave like the old one.
