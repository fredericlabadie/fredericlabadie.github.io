// Case study / Lens 02 / Instrumentation & SDK Debugging

A silent iOS data-loss bug, traced to the initialisation sequence.

// 01 / thesisruntime parity beats documentation parity

Core argument

Telemetry that undercounts silently is worse than telemetry that is missing.

A missing event can be investigated. A silent undercount looks trustworthy while skewing every downstream funnel.

What happened

A cross-platform count gap became an SDK initialisation finding.

A major QSR had a systematic iOS undercount of Application Opened events versus Android. The bug was silent: no visible errors, no obvious dropped-event logs, only a stable undercount. I traced the issue to the app's initialisation sequence and Segment analytics-swift 1.9.1: process(incomingEvent:) returns immediately when enabled == false, so cold-launch events fired before remote config re-enabled the SDK were dropped rather than queued.

The same engagement surfaced broader cross-platform tracking-plan parity gaps and a GTM container governance issue where two international locales were feeding the main analytics container.

Context

The dashboards showed a gap. The instrumentation did not explain it.

Mobile analytics showed a stable difference between iOS and Android Application Opened counts. Downstream funnel reading was being skewed, but there was no explicit error pointing to the source of the loss.

Risk

The undercount looked like real user behaviour.

Because the loss was silent, stakeholders could read the difference as platform behaviour rather than measurement failure. That made root-cause discipline essential: read the app startup sequence and the SDK source path actually running in production.

// 02 / diagnosisstartup order / SDK gate / parity / governance

Diagnosis

The bug lived between startup order and SDK behaviour.

Cross-platform parity failures often only appear at runtime, on cold launch, in one operating system.

1Count gap

Systematic iOS undercount of Application Opened versus Android surfaced in mobile analytics.

2Startup trace

Initialisation sequence showed rawSegment.enabled = false at startup, re-enabled only after remote config completed.

3SDK source read

Segment analytics-swift 1.9.1 returned immediately when disabled, dropping cold-launch events instead of queueing them.

4Governance sweep

Broader review surfaced tracking-plan parity gaps and two international locales feeding the main GTM analytics container.

Action

The fix path required reading what actually shipped.

I traced the undercount through runtime behaviour rather than treating it as a reporting discrepancy. That meant mapping the app's startup order, identifying when Application Opened fired, reading the version-pinned SDK source, and showing why those events were being dropped silently.

// 03 / resultroot cause / parity / governance

Result

The issue moved from mysterious platform difference to explainable data loss.

That distinction matters because the next action changes: do not optimise the funnel until the funnel counts are trustworthy.

Outcome

The root cause became actionable.

  • Silent iOS undercount traced to cold-launch event loss.
  • Segment analytics-swift 1.9.1 initialisation behavior identified as the drop path.
  • Cross-platform tracking-plan parity gaps surfaced in the same engagement.
  • Two international locales found feeding the main analytics GTM container.

Design lesson

Parity is a runtime property.

It is not enough for iOS and Android to share the same tracking plan on paper. If one SDK drops cold-launch events before remote config resolves, the platforms are not equivalent in production, no matter how clean the documentation looks.
