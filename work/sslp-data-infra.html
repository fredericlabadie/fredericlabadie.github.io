The pipeline layer behind the mobile instrumentation.

Scheduled Dataform pipelines transformed raw GA4 events into analysis-ready BigQuery tables feeding production Looker Studio dashboards, with query ownership transferred to the client at rolloff.

// 01 / thesisinstrumentation needs a data layer

Core argument

Raw GA4 events are not an analytics product.

The mobile instrumentation was only useful once the data layer made it analysis-ready and transferable to the client.

What happened

A production pipeline made mobile telemetry usable after rolloff.

A state-regulated retail platform now runs production Looker Studio dashboards, including executive revenue-linked product reporting, on a scheduled Dataform pipeline that transforms raw GA4 events into analysis-ready BigQuery tables.

The production SQL was authored, maintained in the agency-client analytics repo, and formally transferred to the client analytics lead at engagement rolloff so reporting continuity did not depend on the agency retaining query ownership.

Context

The app had instrumentation, but reporting still needed a model.

The client's mobile instrumentation needed an analysis-ready data layer underneath it. Raw GA4 events in BigQuery were not directly usable for executive Looker Studio dashboards. No production pipeline existed for the new app data.

Risk

Reporting would stall if ownership stayed with the agency.

The data layer needed to survive the engagement. That meant production SQL had to live in the repo, be understandable by the client analytics lead, and be transferred before rolloff.

// 02 / pipelineraw GA4 / Dataform / Looker Studio / handoff

Build layer

The dashboard layer depended on three tables and explicit ownership.

The pipeline was not only a transformation job. It was a handoff object.

1GA4_Event_Params

Transforms raw event parameter structures into a table usable for downstream product and event analysis.

2GA4_Event_Users

Produces user-level event context for dashboarding and product reporting.

3GA4_Events

Creates the analysis-ready event table that powers production reporting.

4Rolloff transfer

Query rights formally transferred to the client analytics lead so ownership moved with the system.

Action

Production SQL moved from build artifact to client-owned infrastructure.

I built the scheduled Dataform pipeline producing GA4_Event_Params, GA4_Event_Users, and GA4_Events. I also authored production SQL scripts including bq_product_revenue_inserts.sql and bq_sales_product_data_table_inserts.sql, maintained in the agency-client analytics repo and used for revenue-linked product reporting in executive readouts.

I also scoped a separate product-inventory pipeline modelled against existing product tables, keeping it aligned with the same reporting architecture.

// 03 / resultcontinuity / dashboards / client ownership

Result

The system stayed useful after the engagement ended.

The proof is not only that dashboards ran. It is that the client could own the query layer after rolloff.

Outcome

Reporting continuity was preserved.

  • Production Looker Studio dashboards fed by scheduled Dataform transformations.
  • Executive revenue-linked product reporting powered by authored BigQuery SQL scripts.
  • Agency-client analytics repo used as the working home for production SQL.
  • Query rights transferred to the client analytics lead at rolloff.

Design lesson

Ownership is part of pipeline design.

A pipeline is not complete when it runs. It is complete when the team that needs it can find it, read it, validate it, and operate it without the builder sitting beside them.
