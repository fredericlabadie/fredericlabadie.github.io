Off Adobe, onto Google. Three phases and a cleared backlog.

A D2C health and beverage brand running Adobe Analytics on a Meteor.js SPA had accumulated years of deferred tracking requirements — every change needed heavy developer lift. Three-phase GTM-led migration to GA: port the Adobe implementation, clear the backlog, then add capabilities the platform had never supported.

// 01 / thesisplatform as bottleneck / migration as unlock

Core argument

The analytics platform was the bottleneck, not the tracking requirements.

Adobe Analytics had made every tracking change a developer-heavy operation. The backlog wasn’t a prioritisation problem — it was an architectural one. The migration didn’t just replace the platform; it changed the relationship between analytics requirements and delivery.

What happened

Migration unlocked what the previous platform had made impractical to ship.

A D2C health and beverage brand was running Adobe Analytics on a Meteor.js Single Page Application. Analytics updates required significant developer involvement — tags couldn’t be managed independently — so requirements had accumulated over years without shipping. Xivic was brought in to migrate to GA via GTM. Once the migration was complete, the engagement expanded to clear the backlog and add capabilities Adobe had made impractical: Facebook Ads conversion tracking, JSON-LD SEO schema, and additional enhancement layers the client had not originally scoped.

Context

Meteor.js SPA, Adobe Analytics, and a backlog that Adobe had made impossible to clear.

Meteor.js pre-renders a shell before rendering page content, which meant DOM-ready triggers fired before elements existed. Custom data layer pushes were required to make tracking reliable on the SPA. Adobe Analytics compounded the problem: every tracking change required heavy developer involvement, so requirements sat unshipped. The client wanted a platform that supported independent analytics iteration — changes without a developer ticket for every requirement.

The initial brief was a platform migration. What expanded the engagement was the backlog: once the architecture was in place, years of deferred requirements became tractable.

Task

Migrate to GA/GTM, solve the SPA data layer problem, then work through the backlog.

Full implementation role: reviewing the Adobe Analytics implementation, designing the GTM data layer architecture for Meteor.js, writing and building tags, migrating applicable Adobe components, clearing the backlog across three phases, and shipping Phase 3 enhancements the client had not originally specified. The Meteor.js data layer architecture was the critical design decision — it determined whether all subsequent phases would require developer involvement or not.

// 02 / approachPhase 1: migrate / Phase 2: backlog / Phase 3: enhance

Method

Three phases, each removing a constraint the previous one exposed.

Phase 1 created the foundation. Phase 2 cleared what had accumulated. Phase 3 added capabilities that Adobe had made impractical — things the client had not originally scoped because they assumed the platform couldn’t support rapid iteration.

Instrumentation note

Meteor.js pre-render required a data layer push approach.

Standard GTM trigger logic — fire on DOM ready — was unreliable on this SPA because the pre-render loaded a shell before the actual page content. The solution was to push custom events and variables from the application to the data layer, so GTM triggered only after page content was confirmed loaded. This required close coordination with developers to define the push events and variable schema.

Developer involvement dropped out of the tracking loop for the majority of requirements. The data layer handled event collection through GTM; analytics requirements no longer needed an engineering ticket. The operating model shifted, not just the platform.

// 03 / resultmigration complete / platform independent

Result

The platform now supported independent analytics iteration.

All three phases told the same story: the platform was the constraint, not the requirements. Once the constraint was removed, the backlog cleared and new capabilities followed without a developer queue.

Outcome

Migration complete, backlog cleared, platform enables self-service going forward.

  • Platform migration: Adobe Analytics replaced by GA with full feature parity; data layer architecture in place.
  • SPA handling: Meteor.js pre-render problem solved via data layer push; pageview and event reliability confirmed.
  • Backlog cleared: deferred AdWords tracking and partially-implemented technologies resolved in Phase 2.
  • New capabilities: Facebook conversion tracking, JSON-LD schema, and additional enhancements shipped in Phase 3 — none requiring heavy developer involvement.
  • Operating model change: analytics team no longer dependent on development cadence for the majority of tracking updates.

Design lesson

Platform lock-in isn’t just about the platform. It’s about what the platform prevents.

The most significant cost of the Adobe implementation was not the tool itself — it was the accumulated backlog of requirements that couldn’t ship because every change was too expensive. The migration didn’t just replace the platform; it changed the operational relationship between analytics requirements and delivery.

The three-phase structure mattered: migrating to GA before clearing the backlog avoided the trap of trying to migrate and add capabilities simultaneously. Migrate first, clear second, enhance third is a more maintainable sequence than attempting all three at once.
