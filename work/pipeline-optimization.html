Case study · Topic 07 · Data Engineering & Architecture

Three raw scans → one. Fifteen CTEs → zero. ~50% cost reduction.

An audit and architectural redesign of a GA4 → BigQuery pipeline that had grown organically into a 1.2 TB intermediate table and a $112/run cascading CTE chain.

/ State-regulated retail platform · analytics architect · April 2026

Pipeline debt accumulates the same way technical debt does — one reasonable decision at a time, each solving an immediate problem without visibility into the cumulative architecture. The 1.2 TB GA4_Event_Params table was a reasonable intermediate step when first introduced. The 15-CTE PDP extraction was a reasonable way to get the data. The problem was that no one had stepped back to look at the whole system.
Where the money was going
The original pipeline scanned events_* three separate times (200–400 GB each at $6.25/TB). Step 02.25 unnested every event type into a generic key-value table (GA4_Event_Params — 1.2 TB, 4.67 billion rows). Step 02.5 then ran 15+ separate CTEs each independently scanning that full 1.2 TB table and FULL OUTER JOIN-ed them together: ~18 TB processed per run, the primary cost driver at ~$112/run. On top of that: SELECT DISTINCT on 30–50 columns in three steps; channel-grouping CASE logic duplicated and slightly inconsistent across steps; a 429 GB unpartitioned table forcing full scans on every downstream query.
What replaced it
Single raw event scan via conditional aggregation — one GROUP BY with MAX(IF(ep.key = 'x', ep.value.string_value, NULL)) for each of 25+ event parameter keys simultaneously, producing a flat analysis-ready row per page_view event with all parameters pivoted into columns. 8-step sequential pipeline restructured as a 6-step DAG with three tiers of parallel execution. SELECT DISTINCT replaced with ROW_NUMBER dedup and GROUP BY. Channel grouping unified into a single canonical CASE definition. Partitioning added to the largest reporting tables, converting full-table scans into pruned daily reads.

Outcome
A pipeline that's cheaper, faster, easier to maintain, and more consistent in its output. Migration plan delivered as a 4-phase rollout covering schema preparation, historical backfill, scheduled-query deployment, and table cleanup with a 2–4 week validation window. The optimisation here wasn't about writing faster queries — it was about understanding the full data flow, identifying where costs were actually coming from, and redesigning around a single shared computation rather than repeated independent ones.
