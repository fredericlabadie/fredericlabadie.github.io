The measurement plan written before the code.

// 01 / thesismeasurement before launch

Core argument

Without a measurement plan, a product launch can fool its own team.

Baselines, event coverage, success criteria, and confounds have to be named before the redesign ships, not after the data starts telling convenient stories.

What happened

A planned cutover became a controlled experiment.

A major quick-service restaurant brand was preparing a significant mobile navigation redesign as a straight cutover. I proposed and owned the release measurement and A/B framework before development began: event tagging requirements, tracking plan, and an A/B-rationale one-pager that isolated the navigation change as the causal variable.

The architectural deliverable was the framework itself. Specs, tracking plan, and rationale were handed to engineering and the client analytics lead. Test ownership and rollout decision stayed with the client team. Post-launch readings and the eventual rollout decision remain confidential.

Situation

The redesign was too large to evaluate casually.

The mobile app was heading into a significant navigation redesign, initially planned as a straight cutover. Baselines had not been captured, the right events were not yet instrumented for the new information architecture, and there was no shared contract for what would count as success or failure.

Risk

A concurrent release could contaminate the readout.

An Upsells v1.0 release was running concurrently, raising the risk that post-launch changes would be attributed to the navigation redesign even if another release had influenced the result. The rationale needed to call that out before the test ran.

// 02 / frameworkevents / metrics / guardrails / rationale

Framework

The deliverable was a decision contract.

The framework defined what should be instrumented, what should be watched, and what could invalidate a clean interpretation.

1Event tagging

Specified new events for tab taps, nav state changes, and back-navigation patterns.

2Required properties

Required tab_name, previous_tab, and platform, plus screen_view audit against the new nav destinations.

3Metrics + guardrails

Primary transaction/North Star metric, funnel metrics, nav-specific metrics, and guardrails for session length, crash rate, and error rate.

4A/B rationale

One-pager defined causal isolation, confounding risk, statistical threshold, and minimum run duration before the release.

Action

Two written deliverables moved the release from opinion to evaluation.

The first deliverable was an engineering-ready event-tagging spec with Jira acceptance criteria tied to dev tickets. The second was a release tracking plan plus A/B-rationale one-pager covering primary metrics, funnel metrics, navigation-specific signals, guardrails, statistical confidence, minimum run duration, and the Upsells v1.0 confounding risk.

// 03 / resultcontrolled release / client-owned decision

Result

The result I can show is the test architecture, not the client outcome.

Because post-launch readings are confidential, the case demonstrates the measurement system that made a valid rollout decision possible.

Outcome

The release had rollback options and statistical confidence.

  • Planned straight-cutover reframed into a controlled release.
  • Engineering-ready tagging spec delivered with acceptance criteria.
  • A/B rationale one-pager delivered to engineering and client analytics lead.
  • Confounding risk documented before launch, not rationalised afterward.
  • Client team retained ownership of test execution and rollout decision.

Design lesson

The A/B test is a governance tool.

For a large UX change, an A/B framework is not only a statistical method. It is a governance structure that prevents teams from inventing success criteria after the product has already shipped.
