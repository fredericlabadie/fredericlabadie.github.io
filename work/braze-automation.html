Email telemetry that stops being a copy-paste job.

A Python automation against the Braze REST API that survives campaign edits, clone rebuilds, and exclusion rules because it resolves campaigns dynamically instead of hard-coding IDs.

// 01 / thesisautomation that survives edits

Core argument

The difference between a script and an operating system is whether it survives normal work.

The hard part was not pulling Braze metrics. It was designing the matching rules so the automation kept working after campaigns were cloned, rebuilt, renamed, or excluded.

What happened

Manual monthly extraction became zero-click reporting.

A four-person demand-marketing team went from roughly two hours of manual Braze data extraction per month to zero Braze UI clicks per cycle. A Python automation against the Braze REST API now writes send volume, open rate, click rate, unsubscribe rate, and revenue attribution directly to the team's Google Sheet template.

The key engineering decision was dynamic campaign-name matching with explicit exclusion rules, rather than hard-coded campaign IDs. That is what lets the automation survive campaign clones and rebuilds.

Context

The recurring task was small enough to ignore and frequent enough to matter.

The demand-marketing team spent roughly two hours per month extracting campaign metrics from Braze for the monthly readout. The time cost was not huge, but the task repeated, pulled skilled attention into copy-paste work, and introduced fragility into a recurring reporting cycle.

Risk

Hard-coded campaign IDs would have made the automation brittle.

Campaigns were often cloned or rebuilt. Any automation tied to fixed campaign IDs could silently fail or drift from the intended reporting set. A known-noisy luxury form-submission campaign also needed to be excluded from aggregates without manual triage every cycle.

// 02 / automation designBraze API / matching rules / Google Sheets

Build layer

The durable part was the campaign-resolution logic.

Automation quality depends less on the API call and more on how the script handles normal marketing operations.

1Campaign discovery

Resolve campaigns by name pattern rather than static ID so cloned and rebuilt campaigns continue to map correctly.

2Exclusion rules

Explicitly exclude known-noisy luxury form-submission campaigns from aggregate reporting.

3Metric extraction

Pull send volume, open rate, click rate, unsubscribe rate, and revenue attribution from the Braze REST API.

4Sheet writeback

Write the reporting output directly into the Google Sheet tab the team already used for the monthly readout.

Action

The script was designed around campaign-maintenance reality.

I shipped a Python automation against the Braze REST API with dynamic campaign-ID resolution by name pattern, explicit exclusion rules, and Google Sheets output. The automation writes the recurring readout metrics directly into the team's existing reporting template.

// 03 / resulttime recovered / no UI clicks / durable logic

Result

The win was attention, not just minutes.

Two hours a month is modest. Removing the recurring manual task gave the team one fewer reporting chore to remember, repeat, and verify.

Outcome

Monthly campaign reporting became a generated output.

  • ~2 hours saved per month for the four-person demand-marketing team.
  • Zero Braze UI clicks required per reporting cycle.
  • Five recurring metrics written to the monthly sheet: send volume, open rate, click rate, unsubscribe rate, and revenue attribution.
  • First automated delivery shipped on May 26.
  • Dynamic matching rules reduce the risk of silent report breakage when campaigns are cloned or rebuilt.

Design lesson

Small automations still need product thinking.

A script that saves two hours once is useful. A script that keeps saving those hours after the campaign library changes is an operational asset. The engineering decision that mattered was designing for the way marketers actually maintain campaigns.
