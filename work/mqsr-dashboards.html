A campaign dashboard, a schema decomposition, and a North Star that holds.

Extending an existing Amplitude measurement architecture so campaign performance, platform errors, and product-level revenue questions could be analysed without collapsing into mixed fields.

// 01 / thesisdashboarding exposes schema debt

Core argument

A dashboard can surface more than campaign performance.

The dashboard answered campaign questions, but it also revealed where aggregate views and overloaded properties were hiding operational and revenue signals.

What happened

The campaign view exposed a platform issue and a schema limitation.

A major QSR's Amplitude architecture, anchored on a Transactions North Star, gained a BOGO seasonal-promotion dashboard with deal-conversion funnel, year-over-year comparison, loyalty-acquisition lift analysis, and platform-level error breakdown.

The platform breakdown surfaced an Android-isolated error spike that was invisible in aggregate. The same engagement also produced a schema recommendation: decompose the overloaded products.variant property into five discrete sub-properties, unlocking revenue-by-size, upsell attachment, customisation, and combo-penetration analysis.

Context

The North Star was stable, but the supporting schema was doing too much.

The existing Amplitude architecture used Transactions as the North Star, combining Order Completed, Store Loyalty Checkin Completed, and Receipt Scan Checkin Completed. That was a durable anchor. The problem was downstream detail: campaign, platform, and product-variant questions needed more structured supporting data.

Task

Extend the architecture without breaking the measurement model.

The work was to build a seasonal-promotion dashboard inside the existing Amplitude architecture and identify where the schema needed to be made more granular for future analysis.

// 02 / measurement architecturecampaign / platform / schema

Build layer

Three layers of evidence in one engagement.

Campaign funnel, platform diagnostics, and schema design each answered a different question.

1North Star anchor

Worked within the Transactions framework: order completed, store loyalty check-in, and receipt scan check-in.

2Campaign funnel

Built seasonal-promotion funnel analysis across deal exposure, engagement, conversion, and loyalty acquisition.

3Error breakdown

Added platform-level error visibility so Android-specific issues did not disappear in the aggregate.

4Schema decomposition

Proposed splitting mixed product-variant data into discrete fields for size, upsell, customisation, and combo analysis.

Action

The dashboard became a diagnostic surface.

I built the BOGO seasonal-promotion dashboard with deal-conversion funnel, year-over-year comparison, loyalty-acquisition lift, and platform-level error breakdown. Separately, I proposed decomposing products.variant into five discrete sub-properties so future revenue and product-behaviour analysis would not remain trapped inside one overloaded field.

// 03 / resulterror visibility / richer product analysis

Result

The dashboard answered the campaign question and revealed the next architecture question.

The strongest output was not a single chart. It was clearer visibility into which future questions the current schema could not answer.

Outcome

Campaign, platform, and product questions became more separable.

  • BOGO seasonal-promotion dashboard delivered in Amplitude.
  • Android-isolated error spike surfaced through platform-level breakdown.
  • Five sub-properties proposed for the overloaded products.variant field.
  • Four richer analysis paths unlocked: revenue-by-size, upsell attachment, customisation, and combo penetration.

Design lesson

A stable North Star still needs usable detail underneath.

The Transactions North Star held. The work was making sure the supporting instrumentation could answer the more specific questions teams ask once they trust the top-level metric.
