Evidence paths

Each path answers one line of the role description.

The role brief asks for end-to-end experimentation, product discovery, OKR dashboarding with automation, tracking the product's progress, technical input for analytics engineers, and AI tooling for the analytics craft. The evidence below follows that order.

Path 01 · Experimentation VWO · Google Analytics · placement A/B

The quiz worked. The question was where to put it.

Yubico had a product selection quiz quietly driving purchase intent. Phase 1 GA instrumentation confirmed the conversion signal; Phase 2 VWO A/B testing across four store-page variants found the optimal placement — the winning variant generated roughly twice the pathed CVR of the control.

bol fit · "Conduct experiments end-to-end" is this, almost verbatim — hypothesis, instrumentation, a real four-variant test, a winner called on statistical significance. Later A/B work at Stanley Steemer and a consumer electronics retailer used the same method; this case is the cleanest end-to-end record.

Path 02 · Question discovery Transaction anomaly · frontline staff · shadow IT

The workaround broke. The data showed the gap. Frontline staff knew why.

A transaction anomaly surfaced inside an unrelated lifecycle model. A Sense-Making-informed conversation with floor staff explained why: the designed system had never supported the informal compatibility workflow they depended on. The client shipped a compatibility-checking configurator now central to the business.

bol fit · Product discovery that starts with a transaction anomaly nobody could explain from the dashboard alone, and ends with a shipped product capability.

Path 03 · BI enablement BigQuery · Dataform · Looker Studio

The reporting layer survived the people who built it.

A BigQuery, Dataform, and Looker Studio reporting build replaced recurring manual report pulls, recovered 38 analyst hours per month, and survived client-team turnover through structured handoff — documentation, training, and operating practices designed into the system.

bol fit · The OKR-dashboard-that-survives-turnover problem, solved once already — BigQuery, Dataform, Looker Studio, documented handoff.

Path 04 · Alert engineering Amplitude monitors · checkout telemetry · cart_id

Clean taxonomy, trustworthy alerts, enriched context.

Three layers of remediation on a noisy Amplitude alert channel: redesigned thresholds, a hierarchical error-category taxonomy, and a shipped cart_id enrichment on checkout error events, so failures could be correlated to cart instances.

bol fit · Checkout-event alerting specifically — taxonomy, thresholds, and cart_id context enrichment on live checkout telemetry.

Amplitude monitorsError taxonomy designAlert engineeringCheckout telemetry
Path 05 · Instrumentation GTM · security-constrained architecture

All analytics in the tag manager. Not one line in the site code.

Yubico's e-commerce storefront had a security requirement that banned analytics code from the site codebase entirely. Every tag, variable, and trigger lived in GTM — including handling a purchase funnel that became a Single Page Application mid-build.

bol fit · The clearest existing proof of "technical input to analytics engineers" — a security constraint banned analytics code from the site, and the entire measurement layer had to live in GTM.

GTMEnhanced ecommerceSPA instrumentationSecurity-constrained implementation
Path 06 · AI workflows Claude API · Amplitude · alert triage

Designing AI workflows that don't outsource the thinking.

A production alert-triage workflow compressed 230 raw alerts to 39 posts (83% reduction) while preserving human accountability, and surfaced one critical feature failure that human-only review had missed.

bol fit · Direct precedent for "AI tooling for the analytics craft" — production, not a demo, with a human-judgement gate built in on purpose.
